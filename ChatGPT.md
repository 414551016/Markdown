### Codex 是什麼？
> Codex 是由 OpenAI 推出的一款 AI 助理軟體與通用代理工具（General-purpose Agent）。它與一般 ChatGPT 的最大不同在於，Codex 不僅能回答問題，還能直接動手完成實際工作，例如撰寫文件、製作簡報、生成網站、開發 Apps，甚至直接操作你的電腦。
> 總結來說，Codex 的設計理念是讓人類負責「長度（思考與決策）」，而讓 AI 負責「執行」。它透過將使用者的思考方式轉移給 Agent，達成個人知識資產的自動化運作。

以下是 Codex 的核心特性與功能說明：
- 運作模式與環境：
  - 本地端操作： 不同於 ChatGPT 將內容存在雲端，Codex 是一款安裝在 Windows 或 Mac 電腦上的應用程式。它直接在你的本地電腦上操作，生成的文件也都儲存在你的硬碟中。
  - 多任務並行： Codex 支援同時開啟多個對話（Sessions）與專案，能平行處理不同的任務，且各任務間的記錄互相隔離，避免上下文污染。
  - 專案管理（AGENTS.md）： 使用者可以在專案資料夾中建立 AGENTS.md 檔案作為「操作守則」，讓 AI 在執行任務前先讀取必要的背景資訊與規範。
- 核心功能與技術：
  - 電腦使用（Computer Use）：
    這是 Codex 極其強大的功能，讓 AI 能像真人一樣操作瀏覽器或其他軟體（如 Teams、Excel），執行自動化測試、填寫表單或發送訊息。
  - 技能（Skills）系統：
    使用者可以將標準作業流程（SOP）封裝成「技能」。透過**漸進式披露（Progressive Disclosure）**機制，系統平時只載入技能名稱與描述，僅在觸發時才載入詳細指令，這能確保掛載大量技能時不會塞爆 AI 的思考空間（Context Window）
  - 錄製與重播（Record & Replay）： 這是為了終結複雜提示詞而設計的功能。使用者可以直接操作給 AI 看，Codex 會自動將操作流程分析並儲存為可重複執行的技能檔案。
  - 內建生圖與外掛： 整合了 OpenAI 最強的 Image 2 生圖模型，並支援透過**外掛程式（Plugins）**連接 Gmail、Google Drive、GitHub、Canva 和 Figma 等外部工具
- 與其他工具的差異：
  - 對比 ChatGPT： ChatGPT 偏向雲端對話與知識獲取，Codex 則專精於本地執行與跨軟體的工作流程自動化。
  - 對比 Claude Code： 雖然兩者皆為 Agent 工具，但 Codex 提供了更友好的可視化 UI 介面，適合沒有編程基礎的辦公族群，且支援手機遠端控制電腦幹活。

### Codex和Claude區別？
> Codex 與 Claude Code 分別是由 OpenAI 與 Anthropic 所開發的 AI Agent（智能體）工具。雖然兩者都正從「編程助手」轉向「通用智能體」，但在設計理念、目標族群與功能特色上有顯著的差異。

- 目標族群與使用門檻：
  - Codex： 設計上對零基礎的白領工作者更加友好，提供了直觀的 三欄式視覺化 UI 介面，讓一般用戶能輕鬆上手。
    Codex 官方目前將它定位為可完成 feature development(功能開發)、複雜重構、migration(遷移)、code review(程式碼審查) 與自動化任務的軟體開發代理，並可在 ChatGPT、IDE 與終端機中使用。
  - Claude Code： 自由度更高，但介面與操作邏輯更偏向專業開發者。
    Claude Code 官方則將它描述為可理解整個程式碼庫、跨多個檔案與工具工作，並能建立功能、修復錯誤及自動化開發工作的程式設計助手。
  - Codex 與 Claude Code 的主要差別：
    | 比較項目  | OpenAI Codex                                 | Anthropic Claude Code                       |
    | ----- | -------------------------------------------- | ------------------------------------------- |
    | 開發公司  | OpenAI                                       | Anthropic                                   |
    | 核心定位  | 專門完成軟體工程任務的 Coding Agent                     | 以 Claude 模型為核心的 Agentic Coding Tool         |
    | 主要介面  | ChatGPT、Codex CLI、IDE 擴充套件、雲端環境              | 終端機、VS Code／IDE、Claude Code 環境              |
    | 工作方式  | 接收任務後讀取、修改、執行與測試程式，亦可在雲端平行處理多個任務             | 直接操作本機程式碼庫、執行指令、修改多個檔案及呼叫開發工具               |
    | 多代理能力 | 強調工作樹、雲端環境及多個代理平行處理不同任務                      | 支援子代理、Skills 與動態工作流程，可拆分大型任務                |
    | 專案指令  | 可透過 `AGENTS.md`、Skills、設定檔定義開發規範             | 常透過 `CLAUDE.md`、Skills 與自動記憶保存專案規則          |
    | 生態系整合 | ChatGPT、GitHub、Codex CLI、VS Code、Codex SDK 等 | Claude、Claude Code CLI、VS Code、GitHub、MCP 等 |
    | 適合用途  | 大型重構、Pull Request、測試、程式碼審查、平行代理開發            | 理解大型程式碼庫、互動除錯、架構分析、終端機工作流程                  |

- 操作哲學上的差異：
  - Codex：較像「交付任務給軟體工程代理」你可以對 Codex 下達一個相對完整的工程任務，例如：析這個 GitHub Repository，找出登入模組可能的權限提升漏洞，修正程式碼、補上測試，
    Codex的工作重點通常是：理解任務 → 檢查 Repository → 修改多個檔案 → 執行測試與工具 → 產生可審查的變更 → 並產生 Pull Request 說明。
    Codex特別強調雲端沙盒、工作樹與多代理平行工作，適合把不同 issue、模組或程式碼審查工作分派給多個代理。
  - Claude Code：較像「在終端機中共同工作的資深工程師」。Claude Code 常以互動方式與開發者一起分析問題，例如：先不要修改程式碼。閱讀 authentication 目錄，說明目前驗證流程、信任邊界及可能的攻擊面，提出修正計畫，等我確認後再修改。
    它通常很適合以下流程：閱讀程式碼 → 解釋架構與問題 → 和使用者討論方案 → 修改與測試 → 繼續依結果迭代。
    Claude Code也支援子代理、Skills、專案記憶及 IDE 整合，因此同樣能執行長時間、多步驟與跨檔案的開發任務。
- 模型支持與運作環境：
  - 模型類型： Codex 桌面端僅支持 OpenAI 的 GPT 系列模型（如 GPT-5.5）；Claude Code 則默認使用 Claude 模型，但也允許用戶替換成其他模型。
  - 運作模式： 兩者皆可在本地電腦直接操作文件。不過，Codex 將當前項目資料夾視為**「沙箱」（Sandbox）**作為權限運作的地基，這與 Claude Code 將沙箱視為「額外開啟的保護層」在底層邏輯上有所不同。
- 核心功能特色：
  - Codex 獨有/強項功能：
    - 圖像生成： 內置了 OpenAI 最強的 Image 2 (DALL-E) 生圖模型，可直接在對話中生成產品圖或網頁素材。
    - 錄製與重播 (Record & Replay)： 允許用戶透過「實際操作」示範給 AI 看，自動生成技能（Skill），徹底解決流程難以用文字描述的問題。
    - 移動端控制： 支援透過手機遠端控制電腦上的 Codex 執行任務。
    - 電腦使用 (Computer Use)： 具備自動操作瀏覽器與電腦軟體（如 Teams、Excel）的能力。
    - Codex 是「專用型代理」核心目的為：
      讀取程式碼庫。編輯檔案。執行 shell 指令。執行測試。找出錯誤。完成 Pull Request。執行程式碼審查。自動處理軟體工程工作。
      Anthropic將 Claude 定位為通用 AI 模型家族；Claude Code則是使用 Claude 模型執行軟體開發工作的專用工具。
  - Claude 是「通用型 AI」：
    除了程式設計，也可以：分析文件及資料。撰寫文章與報告。進行研究與推理。解釋理論。處理圖像與文件。協助規劃及知識工作。
  - 共同具備： 兩者皆支援 Skills（技能）、MCP（模型上下文協議）、斜槓命令與遠程操控等進階功能。
- 價格與額度機制：
- 適用場景建議：
  - Codex： 擅長執行類型的工作，適合日常辦公、自動化流程整理、生成圖片與文件彙整。
  - Claude Code： 由於 Claude 模型具備極強的寫作與規劃能力，更適合用於複雜任務的架構設計與內容創作。

## 常用 Skill（技能）
> 定義：
> Skill（技能）不是新的 AI 模型，而是一個可以重複使用的「專業工作流程 + 知識 + 工具 + 腳本」套件，讓 Codex 在特定任務上表現得像專家。
> 也就是說：Skill 並不像 Gmail 外掛一樣都是固定產品。它通常是您自己建立的「工作規則」。Skill 不是增加 AI 智慧，而是增加工作經驗。

Codex會先讀取 Skill 的名稱與說明；工作符合條件時，才載入完整的 SKILL.md。您也可以手動指定某個 Skill。
- Skill 是一個資料夾，至少包含：
  ```
  my-skill/
  └── SKILL.md
  ```
- 也可以另外加入：
  ```
  scripts/       自動執行程式
  references/    參考資料
  assets/        表格或文件範本
  ```

### skill-creator（技能創造者）
> skill-creator 是 Codex內建的系統 Skill之一。
- 用途：協助您建立新的 Skill。
- 您只要告訴它：
  ```
  請幫我建立一個每日工作管理 Skill，可以整理 Gmail、Google Calendar 與待辦事項。
  ```
- 它會詢問：Skill 要完成什麼工作？什麼時候啟動？有哪些禁止事項？是否需要程式或範本？

### 每日工作摘要 Skill
- 建議名稱：
  ```
  daily-work-brief
  ```
- 功能：彙整 Gmail。彙整日曆。找出最重要的三件事。檢查逾期與衝突。產生今日工作摘要。提醒需要回覆的郵件。

#### [Skill 實戰教學，從製作到維護的完整指南(19:53)](https://www.youtube.com/watch?v=PuqX3Kv2ino)
> 這部影片詳細介紹了如何為 AI Agent 打造高效且可維護的 Skill (技能組)，將其視為個人與企業知識轉移的核心載體。作者強調製作技能的關鍵在於釐清意圖與原則，而非死板的 SOP，並透過逆向工程或與 AI 共同發想來提取既有的工作流程。為了確保系統穩定性並節省 Token，開發者應善用漸進式披露機制，將繁雜的細節與格式規範抽離至 References (參考資料) 與 Scripts (腳本) 中。最後，維持技能長久價值的秘訣在於持續的迭代與維護，透過對抗模型的升級與工作流的變動，才能將個人思考邏輯真正轉化為能自動執行的數位資產。

```
my-skill/
  SKILL.md  :這是最主要 skill 的文件
    ---
    name:my-skill
    description:...
    ---
    ## 方法論
      教 agent 怎麼做事...
  references/ :按需載入的參考資料，不是每次都需要的文件
  script/
```
- 漸進式披露（Progressive Disclosure）是 Codex 或 AI Agent 系統中一項關鍵的機制，主要目的是為了優化上下文視窗（context window）的使用效率，確保系統在掛載多個技能（Skill）時，不會因為資訊過載而導致 AI 判斷精準度下降或消耗過多 Token。
  - 第一層：啟動與識別 當 Agent 啟動時，系統並不會一次載入所有技能的完整內容，而僅會載入每個技能的名稱（name）與描述（description）,。這一層大約只佔用 100 個 Token，其目的是讓 AI 知道目前有哪些技能可用，但不會佔用太多記憶空間
  - 第二層：觸發與詳細指令 當對話內容觸發了某個特定技能後，系統才會進一步載入該技能的完整 Markdown 文件（即包含具體方法論的內容）。
  - 第三層：按需載入資源 至於更細節的參考資料（references）執行腳本（scripts），則只在任務執行過程中真正需要時才會被載入。如果該任務沒用到這些資源，它們完全不會佔據 AI 的 context。
- 什麼時候該做一個skill？
  - 重複性高、步驟固定：
    如果同一類型的任務一再出現，且執行流程幾乎相同，導致你必須在不同對話中反覆輸入相同的指令或提示詞時。
  - 包含 AI 不知道的領域知識（Domain Knowledge）：
    就是所謂的「眉角」，例如公司特定的文件格式、專案路徑規範等。這些資訊模型在訓練時無法取得，若不做成 Skill，你每次都得重新交代背景。
  - 出錯成本高，需標準化品質：
    當任務產出若出錯會造成很大麻煩（如需對外交付），或是為了防止 AI 隨意「自由發揮」導致品質不穩時，可用 Skill 強制規範流程。
- 製作 Skill 主要有兩條路徑，分別是**「逆向工程」與「將 AI 作為思維夥伴（Thinking Partner）」**
  - 逆向工程 (Reverse Engineering)：過往的 Session → 請 AI 抽出Pattern → 產出skill草稿
    這是一種從既有的對話記錄或工作流程中萃取技能的方法，特別適合新手啟點。
    - 執行流程： 先開啟一個新的對話視窗，直接與 Agent 協作完成一次任務。過程中可能會出錯或需要調整，直到產出讓你滿意的最終結果為止。
    - 轉換為 Skill： 在任務完成後，不要關閉對話。你可以使用專門的 skill creator 工具，或直接下指令請 AI 將剛才的對話轉化為一份「可重複使用的 Skill」，以確保下次執行相同任務時不會再犯同樣的錯誤。
    - 特點： 這種方式是基於實際成功的執行經驗來撰寫方法論，能有效減少「腦補」規則的情況
  - 思維夥伴 (Thinking Partner / Brain Dump)：你腦中的想法 → 告訴目的 + 流程 → 請他產出skill
    當你腦中已有明確的想法，但還沒有實際操作過，也沒有現成的對話紀錄時，可以採用此路徑。
    - 執行流程： 主動發起需求，告訴 AI 你想要達成的目的、想像中的執行流程以及可能遇到的難點。
    - 轉換為 Skill： 直接請 AI 根據你提供的資訊，整理成一份可讓它後續重複執行該任務的 Skill 文件
    - 關鍵： 在此階段，使用者的任務是釐清並表達清楚自己的意圖、上下文、約束方法與目標，讓 AI 擔任執行者，而你則擔任 AI 的 PM（專案經理）。
  - 錄製與重播 (Record & Replay)：
    這是一種更直觀的「做給 AI 看」的方法。透過錄製你在電腦上的實際操作流程（如：從 Gmail 複製收據並貼上試算表），Codex 會分析錄製內容並自動生成一份 Skill 檔案。
  - 評估驅動開發 (Evaluation Driven Development, EDD)：
    這是一個進階技巧，建議在正式製作 Skill 前，先讓 AI 在沒有任何引導的情況下試做一次任務。觀察它在哪裡卡住或出錯，這些系統缺口才是你真正該手動填補到 Skill 中的規則，而非憑空想像的指令
##### 「逆向工程」（Reverse Engineering）**的典型實作過程：以周會報告為例，具體的做法就是你在週三的早上開了一個新的Session跟你的Agent說。
> 今天下午我要報告給老闆聽。內容包含上週的實作的進度、我預計的一下步，以及專案的卡點。請你連接我的 Google Drive找到工作日誌，進到GitHub查看我過去一週的所有Commit，然後彚整成一份 markdown 文件。
- 啟動協作 Session
  在週三早上，你開啟一個新的對話視窗（Session），並像對待真人助理一樣，清楚交代任務目標
  - 任務背景： 告知 AI 今天下午要向主管報告。
  - 報告內容： 明確要求包含「上週實作進度」、「預計下一步」以及「專案卡點（Blockers）」。
  - 資料來源： 指示 AI 連接你的 Google Drive 讀取工作日誌，並進入 GitHub 查看過去一週的 Commit 記錄。
  - 輸出格式： 要求彙整成一份 Markdown（MD）文件。
- 調校與優化流程：
  在這個過程中，AI 可能會出錯或遇到權限問題。你與 AI 會經歷約一個小時的共同協作，不斷調整指令與修正錯誤，直到產出的最終結果讓你完全滿意為止。
- 將成功經驗轉化為 Skill。
  當看到滿意的結果時，**千萬不要關閉該 Session**，因為這是最珍貴的「方法論」來源，就是你最值錢的 Context。此時你可以：
  - 使用工具： 調用 skill creator 技能。Claude Marketplace 安裝 skill-creator ，直接跟 Claude說：幫我安裝 Anthropic 的 skill creator。
    > 請使用 skill creator skill，把我剛剛的對話，轉成一份可重複使用的 skill。確保下次我有一樣的任務時，我們不會像剛剛那樣不斷調整、出錯。
  - 下達指令： 對 AI 說：「請幫我把剛才的對話轉化為一份可重複使用的 Skill，確保下次執行同樣任務時，我們不會再遇到剛才的錯誤」。
- 設定「漸進式披露」的線索：
  為了讓這個 Skill 之後能被自動觸發，你需要為它撰寫清楚的名稱與描述（Description）。
  - 命名建議： 例如 weekly-report-drafting。
  - 描述寫法： 需包含「做什麼」與「何時用」。例如：「當使用者提到周報、團隊更新或狀態彙總時，從 Google Drive 和 GitHub 彙整數據並撰寫週報」。

為什麼要這樣做？
透過這種「逆向工程」，你是基於實際成功的執行經驗來萃取方法論，而非憑空腦補規則。這能確保 Skill 補足的是系統的「真實缺口」（例如特定格式或權限處理），從而達到標準化的產出品質。

## Codex 的"專案"與"任務"有何差異？各用途如何？
- [專案 Project 是什麼？](#專案-project-是什麼)
- [任務 Task 是什麼？](#任務-Task-是什麼)
- [「普通任務」與「排程任務」也要分清楚](#普通任務與排程任務也要分清楚)
> 在 Codex 中，可以把兩者想成：
> 專案（Project）＝工作空間或資料夾
> 任務（Task）＝在這個工作空間內，交給 Codex 完成的一件具體工作
- 結構大致如下：
  ```
  專案：每日工作管理
  ├── 任務：整理今天的重要 Gmail
  ├── 任務：分析今天的日曆行程
  ├── 任務：產生教授回信草稿
  ├── 任務：製作本週工作報告
  └── 任務：整理業務需求提報
  ```
- 專案與任務的主要差異
  | 比較項目   | 專案 Project           | 任務 Task      |
  | ------ | -------------------- | ------------ |
  | 定位     | 長期工作空間               | 一件具體工作       |
  | 使用期限   | 可持續數週、數月或更久          | 通常完成後即告一段落   |
  | 內容     | 檔案、資料夾、指示、對話、設定      | 指令、處理過程及結果   |
  | 數量關係   | 一個專案包含多個任務           | 一個任務通常屬於某個專案 |
  | 背景資料   | 保存整體背景與長期規則          | 使用專案背景完成當次工作 |
  | 適合用途   | 管理同一主題的全部資料          | 完成具體且可驗收的工作  |
  | 例子     | 「每日工作管理系統」           | 「整理今天的重要郵件」  |
  | 程式設計例子 | 某個 GitHub Repository | 修正登入錯誤、增加測試  |
  | 研究例子   | MIRC701 十篇論文分析       | 比較三篇論文的研究差異  |



### 專案 Project 是什麼？
#### 專案是長期工作的「容器」
> 在 ChatGPT 的一般專案中，專案可以包含對話、檔案與指示；使用桌面版 Codex 時，專案通常也會對應到電腦上的一個資料夾，Codex可以在授權範圍內讀取或修改其中的檔案。
專案主要保存：
- 工作資料夾或程式碼 Repository。
- 專案文件。
- 專案相關對話。
- 固定工作指示。
- Codex 需要遵守的規則。
- 過去執行過的任務。
- 專案使用的 Skill、Plugin 或工具設定。

#### 專案適合長期、同一主題的工作
> 每一個專案都保存自己的資料和背景，避免不同工作的內容混在一起。
例如您可以建立以下專案：
- 01_每日工作管理
- 02_MIRC701論文研究
- 03_博士資格考
- 04_資安研究程式
- 05_黃金果種植管理

#### 專案例子：每日工作管理
> 這些是整個專案都應該遵守的長期規則，而不是只用一次的指令。
- 您可以在電腦建立：
  ```
  每日工作管理/
  ├── AGENTS.md
  ├── 每日工作摘要/
  ├── 業務需求/
  ├── 會議紀錄/
  ├── 週報/
  └── 範本/
      ├── 每日摘要範本.md
      ├── 業務需求提報範本.md
      └── 回信規則.md
  ```
- 其中 AGENTS.md 可以告訴 Codex：
  ```
  1. 使用台灣時間。
  2. 郵件依緊急、重要、一般、參考分類。
  3. 只建立 Gmail 草稿，不可自行寄信。
  4. 不可自行承諾預算或完成日期。
  5. 每日摘要必須列出今日最重要的三件事。
  ```

#### 什麼情況應該建立「新專案」？
> 符合以下情況，適合建立新專案：
- 工作主題完全不同：這些主題的資料、規則與目的不同，應分開建立專案。
  例如：
  ```
  每日行政工作
  MIRC701 論文研究
  黃金果田間管理
  資安程式開發
  ```
- 在 GitHub 使用不同資料夾或 Repository，每個程式碼庫或研究資料夾適合作為獨立專案。
  例如：
  ```
  route-aware-defense/
  reinforcement-learning-lab/
  mirc701-presentation/
  ```
- 需要不同的工作規則，這些規則差異很大，應分開管理。
  例如：
  ```
  工作管理專案：不能自動寄信。
  程式開發專案：修改後必須執行測試。
  論文專案：所有分析必須包含 Why、What、How。
  資安專案：只能測試獲得授權的系統。
  ```




### 任務 Task 是什麼？
#### 任務是一件具體要完成的工作
> 官方建議一個任務最好對應一個連貫的工作單元；若仍是同一問題，留在同一任務中能保留推理與修改歷程，只有工作真正分支時才另外建立任務。
- 例如：這就是一個任務。
  ```
  請整理我今天收到的 Gmail，
  找出必須在今天回覆的郵件，
  並產生回信草稿，但不要寄出。
  ```
- 另一個任務可以是：
  ```
  請查看今天和明天的 Google Calendar，
  找出時間衝突，並列出每場會議需要準備的資料。
  ```
- 任務通常有明確的：
  - 目標。
  - 輸入資料。
  - 執行動作。
  - 產出結果。
  - 完成條件。

#### 一個專案可以有很多任務
> 這些任務共用相同的論文、研究規則與專案背景，但每個任務有不同的具體目標。
- 例如：專案：MIRC701 論文研究
  ```
  任務一：分析王世揚論文的 Why、What、How
  任務二：分析莊輝柱論文
  任務三：比較十篇論文的研究差異
  任務四：建立實驗室研究地圖
  任務五：製作教授報告簡報
  任務六：撰寫逐頁簡報旁白
  ```

#### 什麼情況應該建立「新任務」？
> 符合以下情況，通常只需建立新任務，不必建立新專案：
- 同一專案內有新的工作目標
  - 例如「每日工作管理」專案中：
    ```
    任務一：整理今日 Gmail
    任務二：產生明日行程
    任務三：撰寫本週週報
    ```
- 想讓 Codex 同時處理數件事，Codex可以將代理放在不同執行緒中，並在同一專案下平行工作；Git 專案還可以利用獨立 worktree，避免不同任務直接互相衝突。
  - 例如在同一個程式專案中同時建立：
    ```
    任務一：修正登入錯誤
    任務二：檢查資料庫效能
    任務三：補上 API 測試
    ```
- 希望保留獨立的處理歷程
  - 例如：這樣日後較容易查找結果，也不會讓單一對話過度混亂。
    ```
    一個任務專門分析郵件。
    一個任務專門處理行程。
    一個任務專門整理業務需求。
    ```

### 「普通任務」與「排程任務」也要分清楚
> Codex／ChatGPT 中的任務還可分為兩種。
#### 普通任務
> 由您現在下達指令，Codex執行一次：請整理今天的 Gmail。完成後即結束。

#### 排程任務 Scheduled Task
> 按照固定時間重複執行：每週一至週五上午 7:30，整理重要 Gmail、今日行程與三項優先工作。
- 排程任務可以指定：
  - 使用哪一個專案。
  - 執行提示詞。
  - 執行頻率。
  - 使用哪些 Skill。
  - 是否在獨立 Git worktree 中執行。
  - 執行結果要回到既有任務，還是每次建立新結果。
- 可以這樣理解：
  ```
  專案：每日工作管理
     ↓
  普通任務：今天手動整理一次 Gmail
     ↓
  排程任務：每天上午 7:30 自動整理 Gmail
  ```













## GPT-5.6 模型家族
Sol、Terra、Luna 是 GPT-5.6 模型家族的三個不同層級，它們並不是三個完全不同的 AI，而是針對能力、速度、成本做了不同的最佳化。最簡單的理解方式是：
- ☀️ Sol（太陽）：能力最強
  > 解決最難的編碼、研究、知識工作和高風險專業任務。
- 🌍 Terra（地球）：最均衡
  > 適合日常專業工作的一層。
- 🌙 Luna（月亮）：最快、最省資源
  > 則瞄準高頻、大規模、對成本和速度極度敏感的任務。

官方 API 定價也反映了這種定位：Sol 成本最高、Terra 居中、Luna 最低。
| 項目     | ☀️ Sol          | 🌍 Terra   | 🌙 Luna    |
| ------ | --------------- | ---------- | ---------- |
| 定位     | 旗艦模型            | 均衡模型       | 輕量高速模型     |
| 推理能力   | ⭐⭐⭐⭐⭐           | ⭐⭐⭐⭐☆      | ⭐⭐⭐☆☆      |
| 回應速度   | 較慢              | 中等         | 最快         |
| 成本/API | 最高              | 中等         | 最低         |
| 適合工作   | 複雜研究、程式開發、Agent | 日常工作、寫作、分析 | 快速問答、摘要、分類 |
